# Planner 规则说明

本文档描述了炸弹拆除游戏中各个 Planner 的规则和工作机制。

## 1. BasePlanner（基础规划器）

### 核心功能
- 提供智能体动作管理的基础框架
- 实现空闲智能体检测和动作分配机制

### 空闲智能体检测规则
一个智能体被认为是空闲的条件：
1. 状态为 `room`（在房间内）
2. 没有正在进行的动作（`current_action_in_progress` 为 `False`）
3. 动作队列为空
4. 不在拆弹状态（`defusing_time` 为 0）

### 动作生成规则
- 使用随机策略生成动作
- 如果当前房间有炸弹，优先执行拆弹动作（动作代码：5）
- 否则随机选择静止（动作代码：0）或移动到相邻房间（动作代码：1-4）

### 工作流程
1. 检查所有智能体是否完成动作并更新状态
2. 识别空闲智能体并添加到空闲列表
3. 为空闲智能体分配随机动作
4. 取出动作并更新智能体状态

## 2. SyncPlanner（同步规划器）

### 核心功能
- 继承自 BasePlanner
- 实现同步执行策略，所有智能体必须同时空闲才能分配新动作

### 同步执行规则
- 只有当所有智能体都处于空闲状态时，才为所有智能体安排动作
- 通过 `all_idle` 标志跟踪所有智能体是否同时空闲

### 动作优先级规则
1. **拆弹优先**：如果当前房间有未拆除的炸弹，立即执行拆弹动作
2. **移动优先**：如果当前房间没有炸弹，优先随机移动到相邻房间
3. **静止兜底**：如果没有相邻房间，执行静止动作

## 3. OraclePlanner（大模型规划器）

### 核心功能
- 继承自 SyncPlanner
- 利用大模型生成高级动作指令
- 支持调试模式和真实模型调用

### 工作模式
- **调试模式**：生成模拟的大模型指令进行测试
- **正常模式**：调用配置的大模型生成真实指令

### 动作转换机制
将大模型生成的高级指令转换为环境可执行的低级动作：
- `WAIT {time}` → 静止动作（代码：0）
- `Disposal` → 拆弹动作（代码：5）
- `MOVE {room}` → 移动动作（代码：1-4）

### 动作验证规则
在执行动作前进行合法性验证：
1. **WAIT 动作**：时间必须大于 0 且小于等于 10
2. **Disposal 动作**：
   - 当前房间必须有炸弹
   - 炸弹状态必须大于 0（未拆除）
   - 该房间没有其他智能体正在拆弹
3. **MOVE 动作**：目标房间必须与当前房间相邻

### 智能体命名
为每个智能体分配一个人类可读的名字（Alice, Bob, Charlie 等），便于大模型理解和生成指令。

### 模型配置
支持通过 `model.json` 配置文件设置不同的大模型提供商和参数：
- DeepSeek
- 可配置温度、最大 tokens 等参数

## 4. RoCoPlanner（轮流交流规划器）

### 核心功能
- 继承自 SyncPlanner
- 实现智能体轮流发言的交流机制
- 采用同步计划方式，所有智能体必须同时空闲才能规划

### 交流规则
- **每分钟最大交流次数**：默认值为 6 次
- **每轮每个智能体最大发言次数**：4 次
- **轮流发言机制**：所有空闲智能体轮流发言分享信息
- **总结条件**：每个智能体至少发言一次后才能进行总结规划
- **计划失败处理**：计划失败时让下一个智能体继续发言重新计划

### 观察限制
每个智能体只能看到当前房间的局部信息：
- 当前房间的位置
- 当前房间是否有炸弹
- 相邻房间列表

### 历史记录管理
- **历史发言记录**：保存已完成轮次的发言
- **本轮规划发言记录**：保存当前轮次的发言
- **记录时机**：完成成功规划或到达每轮最大发言次数时添加到历史发言

### 文件保存格式
大模型调用记录文件格式：
```
RoCo_Agent_{name}_{step}_{timestamp}_time{t}.json
```
其中：
- `name`：智能体名称
- `step`：当前时间步
- `timestamp`：时间戳
- `t`：该智能体第几次发言

### 工作流程
1. 检查所有智能体是否同时空闲
2. 重置本轮状态（发言计数、历史记录等）
3. 智能体轮流发言，每次发言后检查：
   - 是否所有智能体至少发言一次
   - 如果是，尝试总结规划
   - 验证计划是否可行
4. 如果制定出可行计划，分配动作
5. 如果达到最大交流次数仍无可行计划，所有智能体静止
6. 将本轮发言添加到历史记录

## 5. AsynPlanner（异步规划器）

### 核心功能
- 继承自 RoCoPlanner
- 实现异步规划策略，根据空闲智能体数量采用不同的规划方式
- 支持单个智能体独自计划和多智能体交流协作

### 规划策略
- **单个空闲智能体**：执行独自计划，使用"PLAN"开头，然后使用"NAME {agent} ACTION {action}"为自己规划动作
- **多个空闲智能体**：执行 RoCo 策略，所有空闲智能体轮流发言分享信息
- **新智能体优先**：如果下一个时间步有新的智能体空闲，则下一轮对话时让新智能体优先进行发言

### 独自计划格式
```
PLAN
NAME {agent} ACTION {action}
```

### 观察限制
每个智能体只能看到当前房间的局部信息：
- 当前房间的位置
- 当前房间是否有炸弹
- 相邻房间列表

### 工作流程
1. 检查所有智能体是否完成动作并更新状态
2. 识别空闲智能体并添加到空闲列表
3. 根据空闲智能体数量选择规划策略：
   - **单个空闲智能体**：执行独自计划
   - **多个空闲智能体**：执行 RoCo 策略，优先让新智能体发言
4. 取出动作并更新智能体状态

## 6. CoMAPPlanner（记忆增强规划器）

### 核心功能
- 继承自 AsynPlanner
- 实现时间窗口机制，仅使用最近 K 轮的发言记录
- 引入共享记忆功能，定期更新记忆内容
- 支持单个智能体独自计划和多智能体交流协作

### 时间窗口机制
- 仅使用最近 K 轮的发言记录（默认 K=3）
- 减少上下文长度，提高模型响应速度

### 记忆功能
- **记忆更新机制**：每发生 M 次独立计划或合作规划时触发更新（默认 M=3）
- **记忆内容**：
  1. 任务信息：目标，进度，探查房间信息
  2. 智能体信息：各个智能体状态
  3. 互动信息：智能体达成的共识、协议规划等
- **记忆生成**：调用大模型根据历史对话和当前环境观察生成
- **记忆记录**：保存记忆生成时间以及相应记忆内容

### 规划策略
- **单个空闲智能体**：执行独自计划，使用"PLAN"开头，然后使用"NAME {agent} ACTION {action}"为自己规划动作
- **多个空闲智能体**：执行 RoCo 策略，所有空闲智能体轮流发言分享信息
- **新智能体优先**：如果下一个时间步有新的智能体空闲，则下一轮对话时让新智能体优先进行发言

### 系统提示增强
- 在 system prompt 中包含共享记忆
- 提供最近 K 轮的发言记录
- 帮助智能体更好地理解当前状态和任务进展

### 工作流程
1. 检查所有智能体是否完成动作并更新状态
2. 识别空闲智能体并添加到空闲列表
3. 根据空闲智能体数量选择规划策略：
   - **单个空闲智能体**：执行独自计划
   - **多个空闲智能体**：执行 RoCo 策略，优先让新智能体发言
4. 取出动作并更新智能体状态
5. 检查是否需要更新记忆：
   - 如果计划次数达到更新间隔，调用大模型生成新的共享记忆
   - 保存记忆到历史记录和文件

## 7. 通用规则

### 动作类型
- `0`: 静止（WAIT）
- `1-4`: 移动到相邻房间（MOVE）
- `5`: 拆弹（Disposal）

### 状态管理
每个智能体维护以下状态：
- `last_action`: 上一次执行的动作
- `current_action`: 当前正在执行的动作
- `current_action_in_progress`: 当前动作是否正在进行
- `action_queue`: 待执行的动作队列

### 空闲检测的关键改进
所有 Planner 都已修复空闲检测逻辑，确保正在拆弹的智能体不会被错误地判断为空闲：
```python
if env_agent_info['status'] == 'room' and not agent_data['current_action_in_progress'] and not agent_data['action_queue'] and env_agent_info.get('defusing_time', 0) == 0:
    # 智能体空闲
```

### 动作完成检测
只有当智能体不在拆弹状态时，才能将当前动作标记为已完成：
```python
if env_agent_info['status'] == 'room' and agent_data['current_action_in_progress'] and env_agent_info.get('defusing_time', 0) == 0:
    # 动作完成，更新状态
```

## 8. 使用场景

- **BasePlanner**: 适用于需要基础随机策略的场景，可作为其他 Planner 的基类
- **SyncPlanner**: 适用于需要同步执行的场景，确保所有智能体协调工作
- **OraclePlanner**: 适用于需要智能决策的场景，利用大模型的推理能力制定策略
- **RoCoPlanner**: 适用于需要多智能体交流协作的场景，模拟人类团队讨论
- **AsynPlanner**: 适用于智能体可能在不同时间成为空闲状态的场景，支持异步协作
- **CoMAPPlanner**: 适用于需要记忆增强和时间窗口管理的场景，提高长期任务的规划效率
